\section{Methods}
\label{sec:methods}

This work studies orchestrated manipulation in a simulated pick-and-place task. The robot is an SO-101 arm operating in an Isaac Sim kitchen scene. Each episode contains three oranges and a plate. The objective is to place all three oranges into the plate. The scene provides privileged simulator state, which is used throughout to evaluate whether subtasks have achieved their intended physical outcome. This avoids confounding the orchestration problem with visual state-estimation errors.

\subsection{LightwheelAI Baseline}

The first experimental setting treats the task as a single full-episode imitation-learning problem. This report refers to this training source as the \textbf{LightwheelAI Baseline} dataset. It is the 60-episode \texttt{leisaac-pick-orange-mimic-v0} dataset published by LightwheelAI, the developers of the upstream \texttt{leisaac} library this project builds on, and serves as the official reference dataset for the task. It contains 60 complete pick-and-place demonstrations, where each episode covers the full sequence of picking the three oranges and placing them in the plate. The demonstrations are monolithic: they are not segmented into subtasks, and the learned policy is expected to execute the full task from the beginning of the episode to the end.

Two policy classes are compared: ACT, an action-chunking transformer policy, and SmolVLA, a Vision-Language-Action model. Both are trained from the same 60-demonstration full-task source, using compatible LeRobot dataset versions. SmolVLA receives a single fixed instruction for the whole episode, so neither model exploits language conditioning in any meaningful way. The comparison serves as a sanity check: if SmolVLA performs no better than ACT under this monolithic formulation, it confirms that the VLA model choice alone does not determine performance in this setting.

\subsection{Subtask Decomposition}

Detecting and responding to intermediate failures requires knowing when one behaviour has ended and whether it succeeded. For this reason, the full task is reformulated as a sequence of shorter subtasks. The main subtasks are grasping a selected orange, lifting it, and placing it into the plate. These subtasks create explicit checkpoints where the execution system can evaluate whether the expected physical outcome has been achieved before committing to the next step.

Targeted grasp prompts use relative spatial labels computed from the current orange layout. In simple configurations, the oranges are labelled left, middle, and right. When two oranges are too close along the main horizontal axis to be reliably distinguished by only left/right position, the secondary horizontal axis is used as a tie-breaker, yielding labels such as top, bottom, top left, bottom left, top right, or bottom right. This keeps the language target meaningful even when the oranges are laterally clustered.

In this formulation, a subtask failure is any case where the intended physical outcome is not reached within the allotted time. This includes failed grasps, an orange falling during lift, unsuccessful placement, or timeout.

An intermediate attempt was made by slicing the LightwheelAI Baseline dataset into shorter subtask segments. This produced a subtask-style dataset, but the original demonstrations always follow the same object order. As a result, the sliced data does not provide a meaningful signal connecting a language instruction to a specific target object: target identity is entangled with progress through the fixed sequence rather than with the instruction itself. This limitation motivated the creation of a custom dataset.

\subsection{Teleop Dataset Construction}

A custom teleoperated dataset was collected to support orchestrated execution. This report refers to the final manually collected training set as \textbf{Teleop}. Unlike the LightwheelAI Baseline dataset, each Teleop episode corresponds to one subtask rather than one complete three-orange trajectory. This makes it possible to discard failed teleoperation attempts, annotate each successful attempt with a subtask instruction, and train the policy on behaviours that can be sequenced by a higher-level controller.

The dataset contains instructions such as ``Grasp left orange'', ``Grasp middle orange'', ``Grasp right orange'', ``Grasp top right orange'', ``Pick it up'', and ``Place it into plate''. The pick order is randomised across episodes. This breaks the fixed-order correlation in the original demonstrations and makes relative-position instructions meaningful: the policy cannot infer the target from task progress alone and must instead learn to associate the instruction with the specified orange.

The raw edited Teleop dataset contains 921 subtask episodes. The final training version used for the evaluated Teleop subtask model removes non-manipulation HOME/reset episodes and appends 20 terminal freeze frames to each retained episode, leaving 846 subtask episodes. These 20 frames correspond to one action chunk in the evaluated setup. They hold the terminal visual observation and robot state, and convert the command into a hold-style action at the end of the subtask. The resulting SmolVLA model is therefore trained as a subtask executor rather than as a monolithic full-task policy: after reaching a subtask-terminal configuration, its learned behaviour should settle instead of immediately continuing into another manipulation phase.

\subsection{Orchestrator Design}

The orchestrator is a finite-state controller wrapped around the VLA policy. The learned policy is only responsible for executing three manipulation prompts: GRASP(\textit{target}), LIFT, and PLACE. The controller chooses the next target, sends the corresponding prompt, monitors whether the expected physical outcome has occurred, and decides whether to advance, retry, redirect, or terminate the episode. Success checks use privileged simulator state so that the experiment evaluates the control structure rather than visual state estimation.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{orchestrator_decision_flow.pdf}
    \caption{Decision logic of the orchestrated evaluator. The VLA is queried only for manipulation subtasks; target selection, local retries, target redirection, and the home-pose reset are handled by the controller.}
    \label{fig:orchestrator_flow}
\end{figure}

At the beginning of an episode, and after each completed placement, the controller enters \texttt{SELECT\_TARGET}. It reads the current orange positions, removes oranges already detected in the plate, assigns relative labels such as left, middle, and right, and selects a target for the next GRASP prompt. If no unplaced oranges remain, the episode is considered complete. If a previous GRASP timed out, the controller stores that orange as the last failed target and avoids selecting it when another unplaced orange is available. This target switch is counted as \textbf{target redirection}.

The three learned subtasks are guarded by explicit physical checks. GRASP is confirmed when the gripper is centred on an unplaced orange, the fingertips are closed, and projected contact force is detected on both gripper contacts for consecutive frames. If the physical check confirms a different orange from the requested one, the controller adapts and treats the actually grasped orange as the active target. LIFT is confirmed when the active orange remains held and rises at least \SI{0.06}{m} above its initial height. PLACE is confirmed when the active orange lies inside the oriented plate bounds, remains stable for consecutive frames, and is released with the gripper open.

Each manipulation state has a timeout: 700 steps for GRASP, 400 for LIFT, and 500 for PLACE. A confirmed GRASP advances to LIFT; a confirmed LIFT advances to PLACE; a confirmed PLACE either returns to \texttt{SELECT\_TARGET} for the next orange or ends the episode once all three oranges are in the plate.

The implementation separates three mechanisms that should not be conflated. \textbf{Local retry} is the true recovery mechanism. If the orange is dropped during LIFT or PLACE, or if LIFT/PLACE times out, the controller preserves the same target and returns directly to GRASP. The original sub-goal is still active, and no target-change reset is inserted.

\textbf{Home-pose reset} is a precondition for reliable target switching, not recovery. When GRASP times out, the controller clears the active target and bypasses the VLA entirely. The arm first moves only the shoulder-lift joint toward the episode-start pose for 40 steps, then interpolates all joints to the episode-start pose for 60 steps. After the reset completes, the policy action queue is cleared and target selection resumes from a state where a new language target is more reliable.

\textbf{Target redirection} happens after a GRASP timeout when another unplaced orange is available. The controller intentionally selects a different orange after the home-pose reset, so this is goal abandonment rather than recovery of the original sub-goal. If no alternative target remains, the controller retries the same orange after the reset. In these experiments, redirection is tracked as a mechanism counter, but oranges are not permanently marked as abandoned in the final episode metrics.

\subsection{Autonomous Data Generation}

The autonomous data generation phase used the orchestrated execution system to collect training data without manual teleoperation. This phase was motivated by the first formal evaluation of the Teleop subtask model: subtasking plus orchestration did not improve full-task performance over the monolithic SmolVLA baseline. The next hypothesis was therefore that the subtask policy needed more data, especially data drawn from the states it visits during autonomous execution rather than only from manual demonstrations.

A full autonomous rollout can contain both failed and successful subtasks. Rather than discarding an entire rollout when a subtask fails, the system records the successful subtask trajectories and discards the failed ones.

This produces an automatically generated dataset of successful subtasks collected under the current policy's own execution distribution. This report refers to the balanced retained subset as \textbf{Auto}. Target redirection plays an important role here: without the ability to pivot to a different orange after a GRASP timeout, the system would need to discard the entire rollout. Redirection allows the system to continue collecting successful subtask data from scenes that would otherwise be wasted.

The autonomous generation run produced subtask episodes that were then filtered to balance the dataset by scene state: the number of episodes collected with one or two oranges already in the plate was equalised against those collected at the start of the episode, so the merged dataset does not over-represent early-scene behaviour. After filtering, 414 Auto episodes were retained and merged with the 846 Teleop training episodes to form \textbf{Teleop+Auto}, a 1260-episode subtask dataset. A new model --- referred to as the Teleop+Auto subtask model --- was trained on this combined dataset.

\subsection{Monotask Fair Comparison}

The Teleop+Auto subtask model cannot be compared only against the LightwheelAI Baseline, because both the dataset and the task formulation changed. To isolate the effect of subtasking more clearly, a second SmolVLA model was trained on the same Teleop+Auto trajectories but treated as a \textbf{monotask} policy. The subtask-specific instructions were replaced by the same fixed prompt used in the monolithic setting, ``Place the orange into plate'', and the model was evaluated without the orchestrator's GRASP/LIFT/PLACE checkpoints.

For both the Teleop and Teleop+Auto monotask/subtask comparisons, every evaluation episode uses the same global budget of 5000 environment steps. The orchestrated subtask controller has additional internal timeouts for GRASP, LIFT, and PLACE, but the total episode horizon is identical to the monotask setting.

This comparison also changes the stopping signal seen during training. In the subtask dataset, each episode ends with the 20-frame freeze tail described above. This teaches the VLA that a language-conditioned subtask should end in a stable hold state. The controller still decides success from privileged simulator state during evaluation, but the learned tail makes it less likely that the policy immediately continues into the next behaviour while the orchestrator is waiting to switch prompts.

For the monotask dataset, the same fragmented trajectories are relabelled with the single full-task prompt and the terminal freeze tails are removed. The model therefore does not learn to stop at local subtask boundaries. Although each training episode is still stored as a fragment, the final state of one subtask is usually a valid starting state for another: a completed grasp leads naturally into lifting, and a lifted orange leads naturally into placing. Under the fixed global prompt, the monotask policy can therefore behave more like a continuous full-task policy stitched together through state observations.

The fair comparison therefore asks whether the same trajectory fragments are more effective as a single fixed-prompt policy or as an orchestrated subtask policy with explicit stop-at-boundary behaviour. It separates the effect of the data source from the effect of subtask decomposition and orchestration, but it should not be interpreted as both models receiving identical temporal supervision.
