\section{Discussion \& Conclusion}
\label{sec:discussion}

The results support one part of the motivation of this project: improving long-horizon manipulation robustness cannot be reduced to changing the policy architecture alone. In the seeded LightwheelAI Baseline, SmolVLA clearly outperforms ACT when both are trained on 60 full-task demonstrations and executed as monotask policies without recovery. However, even the stronger SmolVLA monotask baseline completes only 47/100 episodes, so the task remains far from reliable. This indicates that the main limitation is not only the expressiveness of the low-level policy, but also the structure of the execution problem.

The monotask formulation gives the policy no explicit opportunity to verify that an intermediate physical outcome has been achieved. If a local subtask fails, execution simply continues under the same global prompt, with no checkpoint at which the failure could be caught. One plausible consequence is that an unrecovered failure leaves the scene in a state the policy has rarely seen, from which its behaviour becomes less reliable --- though we treat this as a hypothesis rather than a demonstrated mechanism.

Subtask decomposition addresses this issue by introducing intervention points. A grasp, lift, or placement behaviour can be evaluated before the system commits to the next step. This does not make the low-level policy perfect, but it changes how failures affect the global task. Instead of going undetected for the remainder of the episode, a failure can be detected and handled by the execution system.

The sliced LightwheelAI Baseline experiment shows that subtask boundaries are not sufficient by themselves. For recovery, the system must also be able to choose among alternative valid continuations. In the orange-picking task, this means being able to request a specific target orange. A dataset with a fixed object order does not provide the right supervision for this behaviour, because target identity is entangled with progress through the task. The Teleop dataset was therefore necessary to make relative-position instructions meaningful.

Local retry is the only true recovery mechanism: it returns to GRASP for the same target orange after a slip during LIFT or PLACE, preserving the original sub-goal. Target redirection is goal abandonment, not recovery: after a GRASP timeout, the orchestrator can give up on the current orange and attempt a different one. This prevents dead-end execution by finding an alternative path toward the global objective, but it does not recover the original sub-goal. The system handles local slips via retry and grasp timeouts via redirection, and neither substitutes for a reliably successful subtask policy.

The formal orchestrated evaluation shows that this execution structure is not sufficient to improve complete-task performance. The Teleop subtask model achieved 23 successes in 100 evaluated episodes, or 23.00\%, with an average of 1.74 oranges placed. This improves over the Teleop monotask model, which reached 11/100 successes and mean final occupancy 1.02/3, but it remains below the SmolVLA LightwheelAI Baseline in complete-task success. The result suggests that the benefits of explicit subtask monitoring are outweighed by the low reliability of the learned grasping behaviour, especially as the scene evolves after one or more oranges have already been placed.

The added Auto data did not fix this bottleneck. The Teleop+Auto subtask model achieved 20/100 full completions and a mean final occupancy of 1.51/3, below the Teleop subtask model on both metrics. Its subtask metrics show lower GRASP success in early and mid-scene states and lower PLACE success than the Teleop-only model. The intended data-improvement loop therefore failed in this iteration: adding 414 automatically generated successful subtasks reduced, rather than improved, orchestrated performance.

The Teleop+Auto monotask comparison gives a mixed result. The monotask model trained on the same merged trajectories achieved 24/100 full completions, above the Teleop+Auto subtask model's 20/100. However, its mean final occupancy was 1.34/3, below the subtask model's 1.51/3. This suggests that the orchestrated formulation can still preserve partial progress, but it does not convert that progress into full-task success reliably enough. The comparison should also be interpreted with the different temporal supervision in mind: the subtask policy is trained to settle at local boundaries, whereas the monotask policy is trained without those terminal freeze tails and can keep acting under the same global prompt.

Several limitations remain. First, the success checks use privileged simulator state. This is useful for isolating the recovery problem, but it would need to be replaced or approximated by perception-based success detection for deployment outside simulation. Second, each completed reported evaluation contains 100 episodes from a fixed seed set, so the exact percentages should still be treated as limited-sample estimates. Third, the Auto dataset only contains successful subtasks produced by a weak policy; it may reinforce the policy's narrow visited-state distribution rather than broaden it enough to solve later-scene failures.

Fourth, neither model is trained on complete episodes. The monotask ``full-task'' data is not made of continuous three-orange runs but of the same subtask fragments relabelled with a single prompt; because the terminal state of one subtask is approximately a valid starting state for the next, these stitched sequences partly resemble an uninterrupted trajectory, but the policy never observes one. The Teleop data compounds this: only smooth, precise, retry-free subtasks were retained, so the dataset is diverse at the subtask level yet contains no example of a full successful run, nor of the imperfect mid-episode states a policy actually visits at test time. This coverage gap plausibly limits both the monotask model's ability to chain behaviours over the long horizon and the subtask policy's robustness when it lands outside the curated distribution. Collecting some complete, possibly imperfect, full-task trajectories would be a natural way to close it.

In conclusion, this project frames recovery as an execution-level requirement for VLA-based robotic manipulation, but the implemented recovery system does not yet deliver that robustness. The work shows that subtask decomposition creates useful intervention points and enables retry/redirection logic, but those mechanisms cannot compensate for an insufficiently reliable subtask policy. In the current completed seeded results, Auto data hurts the orchestrated subtask model, and the merged-data monotask/subtask comparison remains mixed rather than clearly validating decomposition. The most important future work is therefore not simply to add more autonomous data, but to improve the quality and coverage of the subtask policy, especially GRASP and PLACE in later scene states, before expecting orchestration to improve full-task success.
